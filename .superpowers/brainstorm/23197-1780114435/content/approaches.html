Discovery System: 3 Approaches

How should the overall agent be structured? This choice shapes the whole project.

A

CLI script + cron + SQLite

A single Python script you run manually or schedule via macOS cron. SQLite stores seen job IDs. Outputs a list of new matches to stdout or a file.

Pros

  • Zero infrastructure — just a file
  • Easy to read and debug
  • No background process needed

Cons

  • Cron setup is manual
  • Hard to extend into auto-apply later
  • No internal orchestration between steps
B

Python daemon + APScheduler + SQLite

A long-running Python process with a built-in daily scheduler. Runs in the background, checks ATSes, filters matches, and notifies you (stdout / email / Slack).

Pros

  • Fully automated — no cron config
  • Easy to add notifications
  • Straightforward to reason about

Cons

  • Must stay running (or use launchd)
  • Still ad hoc — adding apply step means more custom glue
C

Modular agent pipeline ★ Recommended

Discrete pipeline stages: discover → filter → tailor → apply. Each stage is an independent module. A lightweight scheduler triggers the daily run. SQLite tracks state across runs.

Pros

  • Each stage is independently testable
  • Resume tailoring and auto-apply slot in naturally
  • Clean separation between discovery and action
  • Easy to add/swap ATS clients without touching the rest

Cons

  • More upfront structure than a script
  • Slightly more files to navigate
